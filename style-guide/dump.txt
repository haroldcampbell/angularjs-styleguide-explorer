[h2] ## Single Responsibility
[h3] ### Rule of 1 
[h5] Y001
[pr-text] 
[pr-rule]   - Define 1 component per file.
[pr-text]   The following example defines the `app` module and its dependencies, defines a controller, and defines a factory all in the same file.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* avoid */
[pr-text]   angular
[pr-text]       .module('app', ['ngRoute'])
[pr-text]       .controller('SomeController', SomeController)
[pr-text]       .factory('someFactory', someFactory);
[pr-text]     
[pr-text]   function SomeController() { }
[pr-text] 
[pr-text]   function someFactory() { }
[pr-text]   ```
[pr-text] 
[pr-text]   The same components are now separated into their own files.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text] 
[pr-text]   // app.module.js
[pr-text]   angular
[pr-text]       .module('app', ['ngRoute']);
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text] 
[pr-text]   // someController.js
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .controller('SomeController', SomeController);
[pr-text] 
[pr-text]   function SomeController() { }
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text] 
[pr-text]   // someFactory.js
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .factory('someFactory', someFactory);
[pr-text]     
[pr-text]   function someFactory() { }
[pr-text]   ```

[h2] ## IIFE
[h3] ### JavaScript Closures
[h5] Y010
[pr-text] 
[pr-rule]   - Wrap AngularJS components in an Immediately Invoked Function Expression (IIFE). 
[pr-why]   *Why?*: An IIFE removes variables from the global scope. This helps prevent variables and function declarations from living longer than expected in the global scope, which also helps avoid variable collisions.
[pr-text] 
[pr-why]   *Why?*: When your code is minified and bundled into a single file for deployment to a production server, you could have collisions of variables and many global variables. An IIFE protects you against both of these by providing variable scope for each file.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* avoid */
[pr-text]   // logger.js
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .factory('logger', logger);
[pr-text] 
[pr-text]   // logger function is added as a global variable  
[pr-text]   function logger() { }
[pr-text] 
[pr-text]   // storage.js
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .factory('storage', storage);
[pr-text] 
[pr-text]   // storage function is added as a global variable  
[pr-text]   function storage() { }
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /**
[pr-text]    * recommended 
[pr-text]    *
[pr-text]    * no globals are left behind 
[pr-text]    */
[pr-text] 
[pr-text]   // logger.js
[pr-text]   (function() {
[pr-text]       'use strict';
[pr-text] 
[pr-text]       angular
[pr-text]           .module('app')
[pr-text]           .factory('logger', logger);
[pr-text] 
[pr-text]       function logger() { }
[pr-text]   })();
[pr-text] 
[pr-text]   // storage.js
[pr-text]   (function() {
[pr-text]       'use strict';
[pr-text] 
[pr-text]       angular
[pr-text]           .module('app')
[pr-text]           .factory('storage', storage);
[pr-text] 
[pr-text]       function storage() { }
[pr-text]   })();
[pr-text]   ```
[pr-text] 
[pr-text]   - Note: For brevity only, the rest of the examples in this guide may omit the IIFE syntax. 
[pr-text] 
[pr-text]   - Note: IIFE's prevent test code from reaching private members like regular expressions or helper functions which are often good to unit test directly on their own. However you can test these through accessible members or by exposing them through their own component. For example placing helper functions, regular expressions or constants in their own factory or constant.
[pr-text]

[h2] ## Modules
[h3] ### Avoid Naming Collisions
[h5] Y020
[pr-text] 
[pr-rule]   - Use unique naming conventions with separators for sub-modules. 
[pr-why]   *Why?*: Unique names help avoid module name collisions. Separators help define modules and their submodule hierarchy. For example `app` may be your root module while `app.dashboard` and `app.users` may be modules that are used as dependencies of `app`. 
[pr-text] 
[h3] ### Definitions (aka Setters)
[h5] Y021
[pr-text] 
[pr-rule]   - Declare modules without a variable using the setter syntax. 
[pr-why]   *Why?*: With 1 component per file, there is rarely a need to introduce a variable for the module.
[pr-text]   
[pr-text]   ```javascript
[pr-text]   /* avoid */
[pr-text]   var app = angular.module('app', [
[pr-text]       'ngAnimate',
[pr-text]       'ngRoute',
[pr-text]       'app.shared',
[pr-text]       'app.dashboard'
[pr-text]   ]);
[pr-text]   ```
[pr-text] 
[pr-text]   Instead use the simple setter syntax.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text]   angular
[pr-text]       .module('app', [
[pr-text]           'ngAnimate',
[pr-text]           'ngRoute',
[pr-text]           'app.shared',
[pr-text]           'app.dashboard'
[pr-text]       ]);
[pr-text]   ```
[pr-text] 
[h3] ### Getters
[h5] Y022
[pr-text] 
[pr-rule]   - When using a module, avoid using a variable and instead use chaining with the getter syntax.
[pr-why]   *Why?*: This produces more readable code and avoids variable collisions or leaks.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* avoid */
[pr-text]   var app = angular.module('app');
[pr-text]   app.controller('SomeController', SomeController);
[pr-text] 
[pr-text]   function SomeController() { }
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .controller('SomeController', SomeController);
[pr-text] 
[pr-text]   function SomeController() { }
[pr-text]   ```
[pr-text] 
[h3] ### Setting vs Getting
[h5] Y023
[pr-text] 
[pr-rule]   - Only set once and get for all other instances.
[pr-why]   *Why?*: A module should only be created once, then retrieved from that point and after.
[pr-text] 
[pr-text]     - Use `angular.module('app', []);` to set a module.
[pr-text]     - Use `angular.module('app');` to get a module. 
[pr-text] 
[h3] ### Named vs Anonymous Functions
[h5] Y024
[pr-text] 
[pr-rule]   - Use named functions instead of passing an anonymous function in as a callback. 
[pr-why]   *Why?*: This produces more readable code, is much easier to debug, and reduces the amount of nested callback code.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* avoid */
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .controller('Dashboard', function() { })
[pr-text]       .factory('logger', function() { });
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text] 
[pr-text]   // dashboard.js
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .controller('Dashboard', Dashboard);
[pr-text] 
[pr-text]   function Dashboard() { }
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   // logger.js
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .factory('logger', logger);
[pr-text] 
[pr-text]   function logger() { }
[pr-text]   ```
[pr-text]

[h2] ## Controllers
[h3] ### controllerAs View Syntax
[h5] Y030
[pr-text] 
[pr-rule]   - Use the [`controllerAs`](http://www.johnpapa.net/do-you-like-your-angular-controllers-with-or-without-sugar/) syntax over the `classic controller with $scope` syntax. 
[pr-why]   *Why?*: Controllers are constructed, "newed" up, and provide a single new instance, and the `controllerAs` syntax is closer to that of a JavaScript constructor than the `classic $scope syntax`. 
[pr-text] 
[pr-why]   *Why?*: It promotes the use of binding to a "dotted" object in the View (e.g. `customer.name` instead of `name`), which is more contextual, easier to read, and avoids any reference issues that may occur without "dotting".
[pr-text] 
[pr-why]   *Why?*: Helps avoid using `$parent` calls in Views with nested controllers.
[pr-text] 
[pr-text]   ```html
[pr-text]   <!-- avoid -->
[pr-text]   <div ng-controller="Customer">
[pr-text]       {{ name }}
[pr-text]   </div>
[pr-text]   ```
[pr-text] 
[pr-text]   ```html
[pr-text]   <!-- recommended -->
[pr-text]   <div ng-controller="Customer as customer">
[pr-text]       {{ customer.name }}
[pr-text]   </div>
[pr-text]   ```
[pr-text] 
[h3] ### controllerAs Controller Syntax
[h5] Y031
[pr-text] 
[pr-rule]   - Use the `controllerAs` syntax over the `classic controller with $scope` syntax. 
[pr-rule]   - The `controllerAs` syntax uses `this` inside controllers which gets bound to `$scope`
[pr-why]   *Why?*: `controllerAs` is syntactic sugar over `$scope`. You can still bind to the View and still access `$scope` methods.
[pr-text] 
[pr-why]   *Why?*: Helps avoid the temptation of using `$scope` methods inside a controller when it may otherwise be better to avoid them or move them to a factory. Consider using `$scope` in a factory, or if in a controller just when needed. For example when publishing and subscribing events using [`$emit`](https://docs.angularjs.org/api/ng/type/$rootScope.Scope#$emit), [`$broadcast`](https://docs.angularjs.org/api/ng/type/$rootScope.Scope#$broadcast), or [`$on`](https://docs.angularjs.org/api/ng/type/$rootScope.Scope#$on) consider moving these uses to a factory and invoke from the controller. 
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* avoid */
[pr-text]   function Customer($scope) {
[pr-text]       $scope.name = {};
[pr-text]       $scope.sendMessage = function() { };
[pr-text]   }
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended - but see next section */
[pr-text]   function Customer() {
[pr-text]       this.name = {};
[pr-text]       this.sendMessage = function() { };
[pr-text]   }
[pr-text]   ```
[pr-text] 
[h3] ### controllerAs with vm
[h5] Y032
[pr-text] 
[pr-rule]   - Use a capture variable for `this` when using the `controllerAs` syntax. Choose a consistent variable name such as `vm`, which stands for ViewModel.
[pr-why]   *Why?*: The `this` keyword is contextual and when used within a function inside a controller may change its context. Capturing the context of `this` avoids encountering this problem.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* avoid */
[pr-text]   function Customer() {
[pr-text]       this.name = {};
[pr-text]       this.sendMessage = function() { };
[pr-text]   }
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text]   function Customer() {
[pr-text]       var vm = this;
[pr-text]       vm.name = {};
[pr-text]       vm.sendMessage = function() { };
[pr-text]   }
[pr-text]   ```
[pr-text] 
[pr-text]   Note: You can avoid any [jshint](http://www.jshint.com/) warnings by placing the comment below above the line of code. However it is not needed when the function is named using UpperCasing, as this convention means it is a constructor function, which is what a controller is in Angular.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* jshint validthis: true */
[pr-text]   var vm = this;
[pr-text]   ```
[pr-text] 
[pr-text]   Note: When creating watches in a controller using `controller as`, you can watch the `vm.*` member using the following syntax. (Create watches with caution as they add more load to the digest cycle.)
[pr-text] 
[pr-text]   ```html
[pr-text]   <input ng-model="vm.title"/>
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   function SomeController($scope, $log) {
[pr-text]       var vm = this;
[pr-text]       vm.title = 'Some Title';
[pr-text] 
[pr-text]       $scope.$watch('vm.title', function(current, original) {
[pr-text]           $log.info('vm.title was %s', original);
[pr-text]           $log.info('vm.title is now %s', current);
[pr-text]       });
[pr-text]   }
[pr-text]   ```
[pr-text] 
[h3] ### Bindable Members Up Top
[h5] Y033
[pr-text] 
[pr-rule]   - Place bindable members at the top of the controller, alphabetized, and not spread through the controller code.
[pr-why]     *Why?*: Placing bindable members at the top makes it easy to read and helps you instantly identify which members of the controller can be bound and used in the View. 
[pr-text] 
[pr-why]     *Why?*: Setting anonymous functions in-line can be easy, but when those functions are more than 1 line of code they can reduce the readability. Defining the functions below the bindable members (the functions will be hoisted) moves the implementation details down, keeps the bindable members up top, and makes it easier to read. 
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* avoid */
[pr-text]   function Sessions() {
[pr-text]       var vm = this;
[pr-text] 
[pr-text]       vm.gotoSession = function() {
[pr-text]         /* ... */
[pr-text]       };
[pr-text]       vm.refresh = function() {
[pr-text]         /* ... */
[pr-text]       };
[pr-text]       vm.search = function() {
[pr-text]         /* ... */
[pr-text]       };
[pr-text]       vm.sessions = [];
[pr-text]       vm.title = 'Sessions';
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text]   function Sessions() {
[pr-text]       var vm = this;
[pr-text] 
[pr-text]       vm.gotoSession = gotoSession;
[pr-text]       vm.refresh = refresh;
[pr-text]       vm.search = search;
[pr-text]       vm.sessions = [];
[pr-text]       vm.title = 'Sessions';
[pr-text] 
[pr-text]       ////////////
[pr-text] 
[pr-text]       function gotoSession() {
[pr-text]         /* */
[pr-text]       }
[pr-text] 
[pr-text]       function refresh() {
[pr-text]         /* */
[pr-text]       }
[pr-text] 
[pr-text]       function search() {
[pr-text]         /* */
[pr-text]       }
[pr-text]   ```
[pr-text] 
[pr-text]     ![Controller Using "Above the Fold"](https://raw.githubusercontent.com/johnpapa/angularjs-styleguide/master/assets/above-the-fold-1.png)
[pr-text] 
[pr-text]   Note: If the function is a 1 liner consider keeping it right up top, as long as readability is not affected.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* avoid */
[pr-text]   function Sessions(data) {
[pr-text]       var vm = this;
[pr-text] 
[pr-text]       vm.gotoSession = gotoSession;
[pr-text]       vm.refresh = function() {
[pr-text]           /** 
[pr-text]            * lines 
[pr-text]            * of
[pr-text]            * code
[pr-text]            * affects
[pr-text]            * readability
[pr-text]            */
[pr-text]       };
[pr-text]       vm.search = search;
[pr-text]       vm.sessions = [];
[pr-text]       vm.title = 'Sessions';
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text]   function Sessions(dataservice) {
[pr-text]       var vm = this;
[pr-text] 
[pr-text]       vm.gotoSession = gotoSession;
[pr-text]       vm.refresh = dataservice.refresh; // 1 liner is OK
[pr-text]       vm.search = search;
[pr-text]       vm.sessions = [];
[pr-text]       vm.title = 'Sessions';
[pr-text]   ```

[h3] ### Function Declarations to Hide Implementation Details
[h5] Y034
[pr-text] 
[pr-rule]   - Use function declarations to hide implementation details. Keep your bindable members up top. When you need to bind a function in a controller, point it to a function declaration that appears later in the file. This is tied directly to the section Bindable Members Up Top. For more details see [this post](http://www.johnpapa.net/angular-function-declarations-function-expressions-and-readable-code).
[pr-why]     *Why?*: Placing bindable members at the top makes it easy to read and helps you instantly identify which members of the controller can be bound and used in the View. (Same as above.)
[pr-text] 
[pr-why]     *Why?*: Placing the implementation details of a function later in the file moves that complexity out of view so you can see the important stuff up top.
[pr-text] 
[pr-why]     *Why?*: Function declaration are hoisted so there are no concerns over using a function before it is defined (as there would be with function expressions).
[pr-text] 
[pr-why]     *Why?*: You never have to worry with function declarations that moving `var a` before `var b` will break your code because `a` depends on `b`.
[pr-text] 
[pr-why]     *Why?*: Order is critical with function expressions 
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /** 
[pr-text]    * avoid 
[pr-text]    * Using function expressions.
[pr-text]    */
[pr-text]   function Avengers(dataservice, logger) {
[pr-text]       var vm = this;
[pr-text]       vm.avengers = [];
[pr-text]       vm.title = 'Avengers';
[pr-text] 
[pr-text]       var activate = function() {
[pr-text]           return getAvengers().then(function() {
[pr-text]               logger.info('Activated Avengers View');
[pr-text]           });
[pr-text]       }
[pr-text] 
[pr-text]       var getAvengers = function() {
[pr-text]           return dataservice.getAvengers().then(function(data) {
[pr-text]               vm.avengers = data;
[pr-text]               return vm.avengers;
[pr-text]           });
[pr-text]       }
[pr-text] 
[pr-text]       vm.getAvengers = getAvengers;
[pr-text] 
[pr-text]       activate();
[pr-text]   }
[pr-text]   ```
[pr-text] 
[pr-text]   Notice that the important stuff is scattered in the preceding example. In the example below, notice that the important stuff is up top. For example, the members bound to the controller such as `vm.avengers` and `vm.title`. The implementation details are down below. This is just easier to read.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /*
[pr-text]    * recommend
[pr-text]    * Using function declarations
[pr-text]    * and bindable members up top.
[pr-text]    */
[pr-text]   function Avengers(dataservice, logger) {
[pr-text]       var vm = this;
[pr-text]       vm.avengers = [];
[pr-text]       vm.getAvengers = getAvengers;
[pr-text]       vm.title = 'Avengers';
[pr-text] 
[pr-text]       activate();
[pr-text] 
[pr-text]       function activate() {
[pr-text]           return getAvengers().then(function() {
[pr-text]               logger.info('Activated Avengers View');
[pr-text]           });
[pr-text]       }
[pr-text] 
[pr-text]       function getAvengers() {
[pr-text]           return dataservice.getAvengers().then(function(data) {
[pr-text]               vm.avengers = data;
[pr-text]               return vm.avengers;
[pr-text]           });
[pr-text]       }
[pr-text]   }
[pr-text]   ```

[h3] ### Defer Controller Logic
[h5] Y035
[pr-text] 
[pr-rule]   - Defer logic in a controller by delegating to services and factories.
[pr-why]     *Why?*: Logic may be reused by multiple controllers when placed within a service and exposed via a function.
[pr-text] 
[pr-why]     *Why?*: Logic in a service can more easily be isolated in a unit test, while the calling logic in the controller can be easily mocked.
[pr-text] 
[pr-why]     *Why?*: Removes dependencies and hides implementation details from the controller.
[pr-text] 
[pr-text]   ```javascript
[pr-text] 
[pr-text]   /* avoid */
[pr-text]   function Order($http, $q, config, userInfo) {
[pr-text]       var vm = this;
[pr-text]       vm.checkCredit = checkCredit;
[pr-text]       vm.isCreditOk;
[pr-text]       vm.total = 0;
[pr-text] 
[pr-text]       function checkCredit() {
[pr-text]           var settings = {};
[pr-text]           // Get the credit service base URL from config
[pr-text]           // Set credit service required headers
[pr-text]           // Prepare URL query string or data object with request data
[pr-text]           // Add user-identifying info so service gets the right credit limit for this user.
[pr-text]           // Use JSONP for this browser if it doesn't support CORS
[pr-text]           return $http.get(settings)
[pr-text]               .then(function(data) {
[pr-text]                // Unpack JSON data in the response object
[pr-text]                  // to find maxRemainingAmount
[pr-text]                  vm.isCreditOk = vm.total <= maxRemainingAmount
[pr-text]               })
[pr-text]               .catch(function(error) {
[pr-text]                  // Interpret error
[pr-text]                  // Cope w/ timeout? retry? try alternate service?
[pr-text]                  // Re-reject with appropriate error for a user to see
[pr-text]               });
[pr-text]       };
[pr-text]   }
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text]   function Order(creditService) {
[pr-text]       var vm = this;
[pr-text]       vm.checkCredit = checkCredit;
[pr-text]       vm.isCreditOk;
[pr-text]       vm.total = 0;
[pr-text] 
[pr-text]       function checkCredit() { 
[pr-text]          return creditService.isOrderTotalOk(vm.total)
[pr-text]       .then(function(isOk) { vm.isCreditOk = isOk; })
[pr-text]             .catch(showServiceError);
[pr-text]       };
[pr-text]   }
[pr-text]   ```

[h3] ### Keep Controllers Focused
[h5] Y037
[pr-text] 
[pr-rule]   - Define a controller for a view, and try not to reuse the controller for other views. Instead, move reusable logic to factories and keep the controller simple and focused on its view. 
[pr-why]     *Why?*: Reusing controllers with several views is brittle and good end to end (e2e) test coverage is required to ensure stability across large applications.
[pr-text] 
[h3] ### Assigning Controllers
[h5] Y038
[pr-text] 
[pr-rule]   - When a controller must be paired with a view and either component may be re-used by other controllers or views, define controllers along with their routes. 
[pr-text]     Note: If a View is loaded via another means besides a route, then use the `ng-controller="Avengers as vm"` syntax. 
[pr-text] 
[pr-why]     *Why?*: Pairing the controller in the route allows different routes to invoke different pairs of controllers and views. When controllers are assigned in the view using [`ng-controller`](https://docs.angularjs.org/api/ng/directive/ngController), that view is always associated with the same controller.
[pr-text] 
[pr-text]  ```javascript
[pr-text]   /* avoid - when using with a route and dynamic pairing is desired */
[pr-text] 
[pr-text]   // route-config.js
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .config(config);
[pr-text] 
[pr-text]   function config($routeProvider) {
[pr-text]       $routeProvider
[pr-text]           .when('/avengers', {
[pr-text]             templateUrl: 'avengers.html'
[pr-text]           });
[pr-text]   }
[pr-text]   ```
[pr-text] 
[pr-text]   ```html
[pr-text]   <!-- avengers.html -->
[pr-text]   <div ng-controller="Avengers as vm">
[pr-text]   </div>
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text] 
[pr-text]   // route-config.js
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .config(config);
[pr-text] 
[pr-text]   function config($routeProvider) {
[pr-text]       $routeProvider
[pr-text]           .when('/avengers', {

[pr-text]               controller: 'Avengers',
[pr-text]               controllerAs: 'vm'
[pr-text]           });
[pr-text]   }
[pr-text]   ```

[h2] ## Factories
[h3] ### Single Responsibility
[h5] Y050
[pr-text] 
[pr-rule]   - Factories should have a [single responsibility](http://en.wikipedia.org/wiki/Single_responsibility_principle), that is encapsulated by its context. Once a factory begins to exceed that singular purpose, a new factory should be created.
[h3] ### Singletons
[h5] Y051
[pr-text] 
[pr-rule]   - Factories are singletons and return an object that contains the members of the service.
[pr-text]     Note: [All AngularJS services are singletons](https://docs.angularjs.org/guide/services).
[pr-text] 
[h3] ### Accessible Members Up Top
[h5] Y052
[pr-text] 
[pr-rule]   - Expose the callable members of the service (it's interface) at the top, using a technique derived from the [Revealing Module Pattern](http://addyosmani.com/resources/essentialjsdesignpatterns/book/#revealingmodulepatternjavascript). 
[pr-why]     *Why?*: Placing the callable members at the top makes it easy to read and helps you instantly identify which members of the service can be called and must be unit tested (and/or mocked). 
[pr-text] 
[pr-why]     *Why?*: This is especially helpful when the file gets longer as it helps avoid the need to scroll to see what is exposed.
[pr-text] 
[pr-why]     *Why?*: Setting functions as you go can be easy, but when those functions are more than 1 line of code they can reduce the readability and cause more scrolling. Defining the callable interface via the returned service moves the implementation details down, keeps the callable interface up top, and makes it easier to read.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* avoid */
[pr-text]   function dataService() {
[pr-text]     var someValue = '';
[pr-text]     function save() { 
[pr-text]       /* */
[pr-text]     };
[pr-text]     function validate() { 
[pr-text]       /* */
[pr-text]     };
[pr-text] 
[pr-text]     return {
[pr-text]         save: save,
[pr-text]         someValue: someValue,
[pr-text]         validate: validate
[pr-text]     };
[pr-text]   }
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text]   function dataService() {
[pr-text]       var someValue = '';
[pr-text]       var service = {
[pr-text]           save: save,
[pr-text]           someValue: someValue,
[pr-text]           validate: validate
[pr-text]       };
[pr-text]       return service;
[pr-text] 
[pr-text]       ////////////
[pr-text] 
[pr-text]       function save() { 
[pr-text]           /* */
[pr-text]       };
[pr-text] 
[pr-text]       function validate() { 
[pr-text]           /* */
[pr-text]       };
[pr-text]   }
[pr-text]   ```
[pr-text] 
[pr-text]   This way bindings are mirrored across the host object, primitive values cannot update alone using the revealing module pattern
[pr-text] 
[pr-text]     ![Factories Using "Above the Fold"](https://raw.githubusercontent.com/johnpapa/angularjs-styleguide/master/assets/above-the-fold-2.png)
[pr-text] 
[h3] ### Function Declarations to Hide Implementation Details
[h5] Y053
[pr-text] 
[pr-rule]   - Use function declarations to hide implementation details. Keep your accessible members of the factory up top. Point those to function declarations that appears later in the file. For more details see [this post](http://www.johnpapa.net/angular-function-declarations-function-expressions-and-readable-code).
[pr-why]     *Why?*: Placing accessible members at the top makes it easy to read and helps you instantly identify which functions of the factory you can access externally.
[pr-text] 
[pr-why]     *Why?*: Placing the implementation details of a function later in the file moves that complexity out of view so you can see the important stuff up top.
[pr-text] 
[pr-why]     *Why?*: Function declaration are hoisted so there are no concerns over using a function before it is defined (as there would be with function expressions).
[pr-text] 
[pr-why]     *Why?*: You never have to worry with function declarations that moving `var a` before `var b` will break your code because `a` depends on `b`.
[pr-text] 
[pr-why]     *Why?*: Order is critical with function expressions

[pr-text]           // implementation details go here
[pr-text]       };
[pr-text] 
[pr-text]       var getAvengersCast = function() {
[pr-text]          // implementation details go here
[pr-text]       };
[pr-text] 
[pr-text]       var prime = function() {
[pr-text]          // implementation details go here
[pr-text]       };
[pr-text] 
[pr-text]       var ready = function(nextPromises) {
[pr-text]           // implementation details go here
[pr-text]       };
[pr-text] 
[pr-text]       var service = {
[pr-text]           getAvengersCast: getAvengersCast,
[pr-text]           getAvengerCount: getAvengerCount,
[pr-text]           getAvengers: getAvengers,
[pr-text]           ready: ready
[pr-text]       };
[pr-text] 
[pr-text]       return service;
[pr-text]   }

[pr-text]   ```javascript

[h2] ## Data Services
[h3] ### Separate Data Calls
[h5] Y060
[pr-text] 
[pr-rule]   - Refactor logic for making data operations and interacting with data to a factory. Make data services responsible for XHR calls, local storage, stashing in memory, or any other data operations.
[pr-why]     *Why?*: The controller's responsibility is for the presentation and gathering of information for the view. It should not care how it gets the data, just that it knows who to ask for it. Separating the data services moves the logic on how to get it to the data service, and lets the controller be simpler and more focused on the view.
[pr-text] 
[pr-why]     *Why?*: This makes it easier to test (mock or real) the data calls when testing a controller that uses a data service.
[pr-text] 
[pr-why]     *Why?*: Data service implementation may have very specific code to handle the data repository. This may include headers, how to talk to the data, or other services such as $http. Separating the logic into a data service encapsulates this logic in a single place hiding the implementation from the outside consumers (perhaps a controller), also making it easier to change the implementation.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text] 
[pr-text]   // dataservice factory
[pr-text]   angular
[pr-text]       .module('app.core')
[pr-text]       .factory('dataservice', dataservice);
[pr-text] 
[pr-text]   dataservice.$inject = ['$http', 'logger'];
[pr-text] 
[pr-text]   function dataservice($http, logger) {
[pr-text]       return {
[pr-text]           getAvengers: getAvengers
[pr-text]       };
[pr-text] 
[pr-text]       function getAvengers() {
[pr-text]           return $http.get('/api/maa')
[pr-text]               .then(getAvengersComplete)
[pr-text]               .catch(getAvengersFailed);
[pr-text] 
[pr-text]           function getAvengersComplete(response) {
[pr-text]               return response.data.results;
[pr-text]           }
[pr-text] 
[pr-text]           function getAvengersFailed(error) {
[pr-text]               logger.error('XHR Failed for getAvengers.' + error.data);
[pr-text]           }
[pr-text]       }
[pr-text]   }
[pr-text]   ```
[pr-text]     
[pr-text]     Note: The data service is called from consumers, such as a controller, hiding the implementation from the consumers, as shown below.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text] 
[pr-text]   // controller calling the dataservice factory
[pr-text]   angular
[pr-text]       .module('app.avengers')
[pr-text]       .controller('Avengers', Avengers);
[pr-text] 
[pr-text]   Avengers.$inject = ['dataservice', 'logger'];
[pr-text] 
[pr-text]   function Avengers(dataservice, logger) {
[pr-text]       var vm = this;
[pr-text]       vm.avengers = [];
[pr-text] 
[pr-text]       activate();
[pr-text] 
[pr-text]       function activate() {
[pr-text]           return getAvengers().then(function() {
[pr-text]               logger.info('Activated Avengers View');
[pr-text]           });
[pr-text]       }
[pr-text] 
[pr-text]       function getAvengers() {
[pr-text]           return dataservice.getAvengers()
[pr-text]               .then(function(data) {
[pr-text]                   vm.avengers = data;
[pr-text]                   return vm.avengers;
[pr-text]               });
[pr-text]       }
[pr-text]   }      
[pr-text]   ```
[pr-text] 
[h3] ### Return a Promise from Data Calls
[h5] Y061
[pr-text] 
[pr-rule]   - When calling a data service that returns a promise such as $http, return a promise in your calling function too.
[pr-why]     *Why?*: You can chain the promises together and take further action after the data call completes and resolves or rejects the promise.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text] 
[pr-text]   activate();
[pr-text] 
[pr-text]   function activate() {
[pr-text]       /**
[pr-text]        * Step 1
[pr-text]        * Ask the getAvengers function for the
[pr-text]        * avenger data and wait for the promise
[pr-text]        */
[pr-text]       return getAvengers().then(function() {
[pr-text]           /**
[pr-text]            * Step 4
[pr-text]            * Perform an action on resolve of final promise
[pr-text]            */
[pr-text]           logger.info('Activated Avengers View');
[pr-text]       });
[pr-text]   }
[pr-text] 
[pr-text]   function getAvengers() {
[pr-text]         /**
[pr-text]          * Step 2
[pr-text]          * Ask the data service for the data and wait
[pr-text]          * for the promise
[pr-text]          */
[pr-text]         return dataservice.getAvengers()
[pr-text]             .then(function(data) {
[pr-text]                 /**
[pr-text]                  * Step 3
[pr-text]                  * set the data and resolve the promise
[pr-text]                  */
[pr-text]                 vm.avengers = data;
[pr-text]                 return vm.avengers;
[pr-text]         });
[pr-text]   }
[pr-text]   ```
[pr-text]

[h2] ## Directives

[pr-rule]   - Create one directive per file. Name the file for the directive. 
[pr-why]     *Why?*: It is easy to mash all the directives in one file, but difficult to then break those out so some are shared across apps, some across modules, some just for one module. 
[pr-text] 
[pr-why]     *Why?*: One directive per file is easy to maintain.

[pr-text]   /* avoid */

[h3] ### Manipulate DOM in a Directive
[h5] Y072
[pr-text] 
[pr-rule]   - When manipulating the DOM directly, use a directive. If alternative ways can be used such as using CSS to set styles or the [animation services](https://docs.angularjs.org/api/ngAnimate), Angular templating, [`ngShow`](https://docs.angularjs.org/api/ng/directive/ngShow) or [`ngHide`](https://docs.angularjs.org/api/ng/directive/ngHide), then use those instead. For example, if the directive simply hides and shows, use ngHide/ngShow. 
[pr-why]     *Why?*: DOM manipulation can be difficult to test, debug, and there are often better ways (e.g. CSS, animations, templates)
[pr-text] 
[h3] ### Provide a Unique Directive Prefix
[h5] Y073
[pr-text] 
[pr-rule]   - Provide a short, unique and descriptive directive prefix such as `acmeSalesCustomerInfo` which is declared in HTML as `acme-sales-customer-info`.
[pr-why]     *Why?*: The unique short prefix identifies the directive's context and origin. For example a prefix of `cc-` may indicate that the directive is part of a CodeCamper app while `acme-` may indicate a directive for the Acme company. 
[pr-text] 
[pr-text]     Note: Avoid `ng-` as these are reserved for AngularJS directives. Research widely used directives to avoid naming conflicts, such as `ion-` for the [Ionic Framework](http://ionicframework.com/). 
[pr-text] 
[h3] ### Restrict to Elements and Attributes
[h5] Y074
[pr-text] 
[pr-rule]   - When creating a directive that makes sense as a stand-alone element, allow restrict `E` (custom element) and optionally restrict `A` (custom attribute). Generally, if it could be its own control, `E` is appropriate. General guideline is allow `EA` but lean towards implementing as an element when its stand-alone and as an attribute when it enhances its existing DOM element.
[pr-why]     *Why?*: It makes sense.
[pr-text] 
[pr-why]     *Why?*: While we can allow the directive to be used as a class, if the directive is truly acting as an element it makes more sense as an element or at least as an attribute.
[pr-text] 
[pr-text]     Note: EA is the default for AngularJS 1.3 +
[pr-text] 
[pr-text]   ```html
[pr-text]   <!-- avoid -->
[pr-text]   <div class="my-calendar-range"></div>
[pr-text]   ```

[pr-text]         /* */
[pr-text]       }
[pr-text]   }
[pr-text]   ```

[pr-text]   <!-- recommended -->

[pr-text]   <div my-calendar-range></div>
[pr-text]   ```

[h3] ### Directives and ControllerAs
[h5] Y075
[pr-text] 
[pr-rule]   - Use `controller as` syntax with a directive to be consistent with using `controller as` with view and controller pairings.
[pr-why]     *Why?*: It makes sense and it's not difficult.
[pr-text] 
[pr-text]     Note: The directive below demonstrates some of the ways you can use scope inside of link and directive controllers, using controllerAs. I in-lined the template just to keep it all in one place. 
[pr-text] 
[pr-text]     Note: Regarding dependency injection, see [Manually Identify Dependencies](#manual-annotating-for-dependency-injection).
[pr-text] 
[pr-text]     Note: Note that the directive's controller is outside the directive's closure. This style eliminates issues where the injection gets created as unreachable code after a `return`.
[pr-text] 
[pr-text]   ```html
[pr-text]   <div my-example max="77"></div>
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .directive('myExample', myExample);
[pr-text] 
[pr-text]   function myExample() {
[pr-text]       var directive = {
[pr-text]           restrict: 'EA',
[pr-text]           templateUrl: 'app/feature/example.directive.html',
[pr-text]           scope: {
[pr-text]               max: '='
[pr-text]           },
[pr-text]           link: linkFunc,
[pr-text]           controller: ExampleController,
[pr-text]             controllerAs: 'vm',
[pr-text]             bindToController: true // because the scope is isolated
[pr-text]         };
[pr-text] 
[pr-text]       return directive;
[pr-text] 
[pr-text]       function linkFunc(scope, el, attr, ctrl) {
[pr-text]           console.log('LINK: scope.min = %s *** should be undefined', scope.min);
[pr-text]           console.log('LINK: scope.max = %s *** should be undefined', scope.max);
[pr-text]           console.log('LINK: scope.vm.min = %s', scope.vm.min);
[pr-text]           console.log('LINK: scope.vm.max = %s', scope.vm.max);
[pr-text]       }
[pr-text]   }
[pr-text] 
[pr-text]   ExampleController.$inject = ['$scope'];
[pr-text] 
[pr-text]   function ExampleController($scope) {
[pr-text]       // Injecting $scope just for comparison
[pr-text]       var vm = this;
[pr-text] 
[pr-text]       vm.min = 3;
[pr-text] 
[pr-text]       console.log('CTRL: $scope.vm.min = %s', $scope.vm.min);
[pr-text]       console.log('CTRL: $scope.vm.max = %s', $scope.vm.max);
[pr-text]       console.log('CTRL: vm.min = %s', vm.min);
[pr-text]       console.log('CTRL: vm.max = %s', vm.max);
[pr-text]   }
[pr-text]   ```
[pr-text] 
[pr-text]   ```html
[pr-text]   <!-- example.directive.html -->
[pr-text]   <div>hello world</div>
[pr-text]   <div>max={{vm.max}}<input ng-model="vm.max"/></div>
[pr-text]   <div>min={{vm.min}}<input ng-model="vm.min"/></div>
[pr-text]   ```
[pr-text] 
[h5] Y076
[pr-text] 
[pr-rule]   - Use `bindToController = true` when using `controller as` syntax with a directive when you want to bind the outer scope to the directive's controller's scope.
[pr-why]     *Why?*: It makes it easy to bind outer scope to the directive's controller scope.
[pr-text] 
[pr-text]     Note: `bindToController` was introduced in Angular 1.3.0. 
[pr-text] 
[pr-text]   ```html

[h2] ## Resolving Promises for a Controller
[h3] ### Controller Activation Promises
[h5] Y080
[pr-text] 
[pr-rule]   - Resolve start-up logic for a controller in an `activate` function.
[pr-why]     *Why?*: Placing start-up logic in a consistent place in the controller makes it easier to locate, more consistent to test, and helps avoid spreading out the activation logic across the controller.
[pr-text] 
[pr-why]     *Why?*: The controller `activate` makes it convenient to re-use the logic for a refresh for the controller/View, keeps the logic together, gets the user to the View faster, makes animations easy on the `ng-view` or `ui-view`, and feels snappier to the user.
[pr-text] 
[pr-text]     Note: If you need to conditionally cancel the route before you start use the controller, use a [route resolve](#style-y081) instead.
[pr-text]     
[pr-text]   ```javascript
[pr-text]   /* avoid */
[pr-text]   function Avengers(dataservice) {
[pr-text]       var vm = this;
[pr-text]       vm.avengers = [];
[pr-text]       vm.title = 'Avengers';
[pr-text] 
[pr-text]       dataservice.getAvengers().then(function(data) {
[pr-text]           vm.avengers = data;
[pr-text]           return vm.avengers;
[pr-text]       });
[pr-text]   }
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* recommended */
[pr-text]   function Avengers(dataservice) {
[pr-text]       var vm = this;
[pr-text]       vm.avengers = [];
[pr-text]       vm.title = 'Avengers';
[pr-text] 
[pr-text]       activate();
[pr-text] 
[pr-text]       ////////////
[pr-text] 
[pr-text]       function activate() {
[pr-text]           return dataservice.getAvengers().then(function(data) {
[pr-text]               vm.avengers = data;
[pr-text]               return vm.avengers;
[pr-text]           });
[pr-text]       }
[pr-text]   }
[pr-text]   ```
[pr-text] 
[h3] ### Route Resolve Promises
[h5] Y081
[pr-text] 
[pr-rule]   - When a controller depends on a promise to be resolved before the controller is activated, resolve those dependencies in the `$routeProvider` before the controller logic is executed. If you need to conditionally cancel a route before the controller is activated, use a route resolver.
[pr-rule]   - Use a route resolve when you want to decide to cancel the route before ever transitioning to the View.
[pr-why]     *Why?*: A controller may require data before it loads. That data may come from a promise via a custom factory or [$http](https://docs.angularjs.org/api/ng/service/$http). Using a [route resolve](https://docs.angularjs.org/api/ngRoute/provider/$routeProvider) allows the promise to resolve before the controller logic executes, so it might take action based on that data from the promise.
[pr-text] 
[pr-why]     *Why?*: The code executes after the route and in the controller’s activate function. The View starts to load right away. Data binding kicks in when the activate promise resolves. A “busy” animation can be shown during the view transition (via ng-view or ui-view)
[pr-text] 
[pr-text]     Note: The code executes before the route via a promise. Rejecting the promise cancels the route. Resolve makes the new view wait for the route to resolve. A “busy” animation can be shown before the resolve and through the view transition. If you want to get to the View faster and do not require a checkpoint to decide if you can get to the View, consider the [controller `activate` technique](#style-y080) instead.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* avoid */
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .controller('Avengers', Avengers);
[pr-text] 
[pr-text]   function Avengers(movieService) {
[pr-text]       var vm = this;
[pr-text]       // unresolved
[pr-text]       vm.movies;
[pr-text]       // resolved asynchronously
[pr-text]       movieService.getMovies().then(function(response) {
[pr-text]           vm.movies = response.movies;
[pr-text]       });
[pr-text]   }
[pr-text]   ```
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* better */
[pr-text] 
[pr-text]   // route-config.js
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .config(config);
[pr-text] 
[pr-text]   function config($routeProvider) {
[pr-text]       $routeProvider
[pr-text]           .when('/avengers', {
[pr-text]               templateUrl: 'avengers.html',
[pr-text]               controller: 'Avengers',
[pr-text]               controllerAs: 'vm',
[pr-text]               resolve: {
[pr-text]                   moviesPrepService: function(movieService) {
[pr-text]                       return movieService.getMovies();
[pr-text]                   }
[pr-text]               }
[pr-text]           });
[pr-text]   }
[pr-text] 
[pr-text]   // avengers.js
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .controller('Avengers', Avengers);
[pr-text] 
[pr-text]   Avengers.$inject = ['moviesPrepService'];
[pr-text]   function Avengers(moviesPrepService) {
[pr-text]         var vm = this;
[pr-text]         vm.movies = moviesPrepService.movies;
[pr-text]   }
[pr-text]   ```
[pr-text] 
[pr-text]     Note: The example below shows the route resolve points to a named function, which is easier to debug and easier to handle dependency injection.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   /* even better */
[pr-text]

[pr-text]   angular

[h2] ## Manual Annotating for Dependency Injection
[h3] ### UnSafe from Minification
[h5] Y090
[pr-text] 
[pr-rule]   - Avoid using the shortcut syntax of declaring dependencies without using a minification-safe approach.
[pr-why]     *Why?*: The parameters to the component (e.g. controller, factory, etc) will be converted to mangled variables. For example, `common` and `dataservice` may become `a` or `b` and not be found by AngularJS.
[pr-text] 
[pr-text]     ```javascript
[pr-text]     /* avoid - not minification-safe*/
[pr-text]     angular
[pr-text]         .module('app')
[pr-text]         .controller('Dashboard', Dashboard);
[pr-text] 
[pr-text]     function Dashboard(common, dataservice) {
[pr-text]     }
[pr-text]     ```
[pr-text] 
[pr-text]     This code may produce mangled variables when minified and thus cause runtime errors.
[pr-text] 
[pr-text]     ```javascript
[pr-text]     /* avoid - not minification-safe*/
[pr-text]     angular.module('app').controller('Dashboard', d);function d(a, b) { }
[pr-text]     ```
[pr-text] 
[h3] ### Manually Identify Dependencies
[h5] Y091
[pr-text] 
[pr-rule]   - Use `$inject` to manually identify your dependencies for AngularJS components.
[pr-why]     *Why?*: This technique mirrors the technique used by [`ng-annotate`](https://github.com/olov/ng-annotate), which I recommend for automating the creation of minification safe dependencies. If `ng-annotate` detects injection has already been made, it will not duplicate it.
[pr-text] 
[pr-why]     *Why?*: This safeguards your dependencies from being vulnerable to minification issues when parameters may be mangled. For example, `common` and `dataservice` may become `a` or `b` and not be found by AngularJS.
[pr-text] 
[pr-why]     *Why?*: Avoid creating in-line dependencies as long lists can be difficult to read in the array. Also it can be confusing that the array is a series of strings while the last item is the component's function. 
[pr-text] 
[pr-text]     ```javascript
[pr-text]     /* avoid */
[pr-text]     angular
[pr-text]         .module('app')
[pr-text]         .controller('Dashboard', 
[pr-text]             ['$location', '$routeParams', 'common', 'dataservice', 
[pr-text]                 function Dashboard($location, $routeParams, common, dataservice) {}
[pr-text]             ]);
[pr-text]     ```
[pr-text] 
[pr-text]     ```javascript
[pr-text]     /* avoid */
[pr-text]     angular
[pr-text]       .module('app')
[pr-text]       .controller('Dashboard', 
[pr-text]           ['$location', '$routeParams', 'common', 'dataservice', Dashboard]);
[pr-text]       
[pr-text]     function Dashboard($location, $routeParams, common, dataservice) {
[pr-text]     }
[pr-text]     ```
[pr-text] 
[pr-text]     ```javascript
[pr-text]     /* recommended */
[pr-text]     angular
[pr-text]         .module('app')
[pr-text]         .controller('Dashboard', Dashboard);
[pr-text] 
[pr-text]     Dashboard.$inject = ['$location', '$routeParams', 'common', 'dataservice'];
[pr-text] 
[pr-text]     function Dashboard($location, $routeParams, common, dataservice) {
[pr-text]     }
[pr-text]     ```
[pr-text] 
[pr-text]     Note: When your function is below a return statement the $inject may be unreachable (this may happen in a directive). You can solve this by either moving the $inject above the return statement or by using the alternate array injection syntax. 
[pr-text] 
[pr-text]     Note: [`ng-annotate 0.10.0`](https://github.com/olov/ng-annotate) introduced a feature where it moves the `$inject` to where it is reachable.
[pr-text] 
[pr-text]     ```javascript
[pr-text]     // inside a directive definition
[pr-text]     function outer() {
[pr-text]         return {
[pr-text]             controller: DashboardPanel,
[pr-text]         };
[pr-text] 
[pr-text]         DashboardPanel.$inject = ['logger']; // Unreachable
[pr-text]         function DashboardPanel(logger) {
[pr-text]         }
[pr-text]     }
[pr-text]     ```
[pr-text] 
[pr-text]     ```javascript
[pr-text]     // inside a directive definition
[pr-text]     function outer() {
[pr-text]         DashboardPanel.$inject = ['logger']; // reachable
[pr-text]         return {
[pr-text]             controller: DashboardPanel,
[pr-text]         };
[pr-text] 
[pr-text]         function DashboardPanel(logger) {
[pr-text]         }
[pr-text]     }
[pr-text]     ```
[pr-text]

[pr-text] 
[pr-rule]   - Use $inject to manually identify your route resolver dependencies for AngularJS components.
[pr-why]     *Why?*: This technique breaks out the anonymous function for the route resolver, making it easier to read.

[pr-why]     *Why?*: An `$inject` statement can easily precede the resolver to handle making any dependencies minification safe.

[h2] ## Minification and Annotation
[h3] ### ng-annotate
[h5] Y100
[pr-text] 
[pr-rule]   - Use [ng-annotate](//github.com/olov/ng-annotate) for [Gulp](http://gulpjs.com) or [Grunt](http://gruntjs.com) and comment functions that need automated dependency injection using `/** @ngInject */`
[pr-why]     *Why?*: This safeguards your code from any dependencies that may not be using minification-safe practices.
[pr-text] 
[pr-why]     *Why?*: [`ng-min`](https://github.com/btford/ngmin) is deprecated 
[pr-text] 
[pr-text]     >I prefer Gulp as I feel it is easier to write, to read, and to debug.
[pr-text] 
[pr-text]     The following code is not using minification safe dependencies.
[pr-text] 
[pr-text]     ```javascript
[pr-text]     angular
[pr-text]         .module('app')
[pr-text]         .controller('Avengers', Avengers);
[pr-text] 
[pr-text]     /* @ngInject */
[pr-text]     function Avengers(storageService, avengerService) {
[pr-text]         var vm = this;
[pr-text]         vm.heroSearch = '';
[pr-text]         vm.storeHero = storeHero;
[pr-text] 
[pr-text]         function storeHero() {
[pr-text]             var hero = avengerService.find(vm.heroSearch);
[pr-text]             storageService.save(hero.name, hero);
[pr-text]         }
[pr-text]     }
[pr-text]     ```
[pr-text] 
[pr-text]     When the above code is run through ng-annotate it will produce the following output with the `$inject` annotation and become minification-safe.
[pr-text] 
[pr-text]     ```javascript
[pr-text]     angular
[pr-text]         .module('app')
[pr-text]         .controller('Avengers', Avengers);
[pr-text] 
[pr-text]     /* @ngInject */
[pr-text]     function Avengers(storageService, avengerService) {
[pr-text]         var vm = this;
[pr-text]         vm.heroSearch = '';
[pr-text]         vm.storeHero = storeHero;
[pr-text] 
[pr-text]         function storeHero() {
[pr-text]             var hero = avengerService.find(vm.heroSearch);
[pr-text]             storageService.save(hero.name, hero);
[pr-text]         }
[pr-text]     }
[pr-text] 
[pr-text]     Avengers.$inject = ['storageService', 'avengerService'];
[pr-text]     ```
[pr-text] 
[pr-text]     Note: If `ng-annotate` detects injection has already been made (e.g. `@ngInject` was detected), it will not duplicate the `$inject` code.
[pr-text] 
[pr-text]     Note: When using a route resolver you can prefix the resolver's function with `/* @ngInject */` and it will produce properly annotated code, keeping any injected dependencies minification safe.
[pr-text] 
[pr-text]     ```javascript
[pr-text]     // Using @ngInject annotations
[pr-text]     function config($routeProvider) {
[pr-text]         $routeProvider
[pr-text]             .when('/avengers', {
[pr-text]                 templateUrl: 'avengers.html',
[pr-text]                 controller: 'Avengers',
[pr-text]                 controllerAs: 'vm',
[pr-text]                 resolve: { /* @ngInject */
[pr-text]                     moviesPrepService: function(movieService) {
[pr-text]                         return movieService.getMovies();
[pr-text]                     }
[pr-text]                 }
[pr-text]             });
[pr-text]     }
[pr-text]     ```
[pr-text] 
[pr-text]     > Note: Starting from AngularJS 1.3 use the [`ngApp`](https://docs.angularjs.org/api/ng/directive/ngApp) directive's `ngStrictDi` parameter. When present the injector will be created in "strict-di" mode causing the application to fail to invoke functions which do not use explicit function annotation (these may not be minification safe). Debugging info will be logged to the console to help track down the offending code.
[pr-text]     `<body ng-app="APP" ng-strict-di>`
[pr-text] 
[h3] ### Use Gulp or Grunt for ng-annotate
[h5] Y101
[pr-text] 
[pr-rule]   - Use [gulp-ng-annotate](https://www.npmjs.org/package/gulp-ng-annotate) or [grunt-ng-annotate](https://www.npmjs.org/package/grunt-ng-annotate) in an automated build task. Inject `/* @ngInject */` prior to any function that has dependencies.

[h2] ## Exception Handling
[h3] ### decorators
[h5] Y110
[pr-text] 
[pr-rule]   - Use a [decorator](https://docs.angularjs.org/api/auto/service/$provide#decorator), at config time using the [`$provide`](https://docs.angularjs.org/api/auto/service/$provide) service, on the [`$exceptionHandler`](https://docs.angularjs.org/api/ng/service/$exceptionHandler) service to perform custom actions when exceptions occur.
[pr-why]     *Why?*: Provides a consistent way to handle uncaught AngularJS exceptions for development-time or run-time.
[pr-text] 
[pr-text]     Note: Another option is to override the service instead of using a decorator. This is a fine option, but if you want to keep the default behavior and extend it a decorator is recommended.
[pr-text] 
[pr-text]     ```javascript
[pr-text]     /* recommended */
[pr-text]     angular
[pr-text]         .module('blocks.exception')
[pr-text]         .config(exceptionConfig);
[pr-text] 
[pr-text]     exceptionConfig.$inject = ['$provide'];
[pr-text] 
[pr-text]     function exceptionConfig($provide) {
[pr-text]         $provide.decorator('$exceptionHandler', extendExceptionHandler);
[pr-text]     }
[pr-text] 
[pr-text]     extendExceptionHandler.$inject = ['$delegate', 'toastr'];
[pr-text] 
[pr-text]     function extendExceptionHandler($delegate, toastr) {
[pr-text]         return function(exception, cause) {
[pr-text]             $delegate(exception, cause);
[pr-text]             var errorData = { 
[pr-text]                 exception: exception, 
[pr-text]                 cause: cause 
[pr-text]             };
[pr-text]             /**
[pr-text]              * Could add the error to a service's collection,
[pr-text]              * add errors to $rootScope, log errors to remote web server,
[pr-text]              * or log locally. Or throw hard. It is entirely up to you.
[pr-text]              * throw exception;
[pr-text]              */
[pr-text]             toastr.error(exception.msg, errorData);
[pr-text]         };
[pr-text]     }
[pr-text]     ```
[pr-text] 
[h3] ### Exception Catchers
[h5] Y111
[pr-text] 
[pr-rule]   - Create a factory that exposes an interface to catch and gracefully handle exceptions.
[pr-why]     *Why?*: Provides a consistent way to catch exceptions that may be thrown in your code (e.g. during XHR calls or promise failures).
[pr-text] 
[pr-text]     Note: The exception catcher is good for catching and reacting to specific exceptions from calls that you know may throw one. For example, when making an XHR call to retrieve data from a remote web service and you want to catch any exceptions from that service and react uniquely.
[pr-text] 
[pr-text]     ```javascript
[pr-text]     /* recommended */
[pr-text]     angular
[pr-text]         .module('blocks.exception')
[pr-text]         .factory('exception', exception);
[pr-text] 
[pr-text]     exception.$inject = ['logger'];
[pr-text] 
[pr-text]     function exception(logger) {
[pr-text]         var service = {
[pr-text]             catcher: catcher
[pr-text]         };
[pr-text]         return service;
[pr-text] 
[pr-text]         function catcher(message) {
[pr-text]             return function(reason) {
[pr-text]                 logger.error(message, reason);
[pr-text]             };
[pr-text]         }
[pr-text]     }
[pr-text]     ```
[pr-text] 
[h3] ### Route Errors
[h5] Y112
[pr-text] 
[pr-rule]   - Handle and log all routing errors using [`$routeChangeError`](https://docs.angularjs.org/api/ngRoute/service/$route#$routeChangeError).
[pr-why]     *Why?*: Provides a consistent way handle all routing errors.
[pr-text] 
[pr-why]     *Why?*: Potentially provides a better user experience if a routing error occurs and you route them to a friendly screen with more details or recovery options.
[pr-text] 
[pr-text]     ```javascript
[pr-text]     /* recommended */
[pr-text]     function handleRoutingErrors() {
[pr-text]         /**
[pr-text]          * Route cancellation:
[pr-text]          * On routing error, go to the dashboard.
[pr-text]          * Provide an exit clause if it tries to do it twice.
[pr-text]          */
[pr-text]         $rootScope.$on('$routeChangeError',
[pr-text]             function(event, current, previous, rejection) {
[pr-text]                 var destination = (current && (current.title || current.name || current.loadedTemplateUrl)) ||
[pr-text]                     'unknown target';
[pr-text]                 var msg = 'Error routing to ' + destination + '. ' + (rejection.msg || '');
[pr-text]                 /**
[pr-text]                  * Optionally log using a custom service or $log.
[pr-text]                  * (Don't forget to inject custom service)
[pr-text]                  */
[pr-text]                 logger.warning(msg, [current]);
[pr-text]             }
[pr-text]         );
[pr-text]     }
[pr-text]     ```

[h2] ## Naming
[h3] ### Naming Guidelines
[h5] Y120
[pr-text] 
[pr-rule]   - Use consistent names for all components following a pattern that describes the component's feature then (optionally) its type. My recommended pattern is `feature.type.js`. There are 2 names for most assets:
[pr-text]     * the file name (`avengers.controller.js`)
[pr-text]     * the registered component name with Angular (`AvengersController`)
[pr-text]  
[pr-why]     *Why?*: Naming conventions help provide a consistent way to find content at a glance. Consistency within the project is vital. Consistency with a team is important. Consistency across a company provides tremendous efficiency.
[pr-text] 
[pr-why]     *Why?*: The naming conventions should simply help you find your code faster and make it easier to understand.

[h3] ### Modules
[h5] Y127
[pr-text] 
[pr-rule]   - When there are multiple modules, the main module file is named `app.module.js` while other dependent modules are named after what they represent. For example, an admin module is named `admin.module.js`. The respective registered module names would be `app` and `admin`. 
[pr-why]     *Why?*: Provides consistency for multiple module apps, and for expanding to large applications.
[pr-text] 
[pr-why]     *Why?*: Provides easy way to use task automation to load all module definitions first, then all other angular files (for bundling).
[pr-text] 
[h3] ### Configuration
[h5] Y128
[pr-text] 
[pr-rule]   - Separate configuration for a module into its own file named after the module. A configuration file for the main `app` module is named `app.config.js` (or simply `config.js`). A configuration for a module named `admin.module.js` is named `admin.config.js`.
[pr-why]     *Why?*: Separates configuration from module definition, components, and active code.
[pr-text] 
[pr-why]     *Why?*: Provides a identifiable place to set configuration for a module.
[pr-text] 
[h3] ### Routes
[h5] Y129
[pr-text] 
[pr-rule]   - Separate route configuration into its own file. Examples might be `app.route.js` for the main module and `admin.route.js` for the `admin` module. Even in smaller apps I prefer this separation from the rest of the configuration.

[h2] ## Application Structure LIFT Principle
[h3] ### LIFT
[h5] Y140
[pr-text] 
[pr-rule]   - Structure your app such that you can `L`ocate your code quickly, `I`dentify the code at a glance, keep the `F`lattest structure you can, and `T`ry to stay DRY. The structure should follow these 4 basic guidelines. 
[pr-text]     *Why LIFT?*: Provides a consistent structure that scales well, is modular, and makes it easier to increase developer efficiency by finding code quickly. Another way to check your app structure is to ask yourself: How quickly can you open and work in all of the related files for a feature?
[pr-text] 
[pr-text]     When I find my structure is not feeling comfortable, I go back and revisit these LIFT guidelines
[pr-text]   
[pr-text]     1. `L`ocating our code is easy
[pr-text]     2. `I`dentify code at a glance
[pr-text]     3. `F`lat structure as long as we can
[pr-text]     4. `T`ry to stay DRY (Don’t Repeat Yourself) or T-DRY
[pr-text] 
[h3] ### Locate
[h5] Y141
[pr-text] 
[pr-rule]   - Make locating your code intuitive, simple and fast.
[pr-why]     *Why?*: I find this to be super important for a project. If the team cannot find the files they need to work on quickly, they will not be able to work as efficiently as possible, and the structure needs to change. You may not know the file name or where its related files are, so putting them in the most intuitive locations and near each other saves a ton of time. A descriptive folder structure can help with this.
[pr-text] 
[pr-text]     ```
[pr-text]     /bower_components
[pr-text]     /client
[pr-text]       /app
[pr-text]         /avengers
[pr-text]         /blocks
[pr-text]           /exception
[pr-text]           /logger
[pr-text]         /core
[pr-text]         /dashboard
[pr-text]         /data
[pr-text]         /layout
[pr-text]         /widgets
[pr-text]       /content
[pr-text]       index.html
[pr-text]     .bower.json
[pr-text]     ```
[pr-text] 
[h3] ### Identify
[h5] Y142
[pr-text] 
[pr-rule]   - When you look at a file you should instantly know what it contains and represents.
[pr-why]     *Why?*: You spend less time hunting and pecking for code, and become more efficient. If this means you want longer file names, then so be it. Be descriptive with file names and keeping the contents of the file to exactly 1 component. Avoid files with multiple controllers, multiple services, or a mixture. There are deviations of the 1 per file rule when I have a set of very small features that are all related to each other, they are still easily identifiable.
[pr-text] 
[h3] ### Flat
[h5] Y143
[pr-text] 
[pr-rule]   - Keep a flat folder structure as long as possible. When you get to 7+ files, begin considering separation.
[pr-why]     *Why?*: Nobody wants to search 7 levels of folders to find a file. Think about menus on web sites … anything deeper than 2 should take serious consideration. In a folder structure there is no hard and fast number rule, but when a folder has 7-10 files, that may be time to create subfolders. Base it on your comfort level. Use a flatter structure until there is an obvious value (to help the rest of LIFT) in creating a new folder.
[pr-text] 
[h3] ### T-DRY (Try to Stick to DRY)
[h5] Y144
[pr-text] 
[pr-rule]   - Be DRY, but don't go nuts and sacrifice readability.
[pr-why]     *Why?*: Being DRY is important, but not crucial if it sacrifices the others in LIFT, which is why I call it T-DRY. I don’t want to type session-view.html for a view because, well, it’s obviously a view. If it is not obvious or by convention, then I name it. 
[pr-text] 
[pr-text] 
[h2] ## Application Structure
[h3] ### Overall Guidelines
[h5] Y150
[pr-text] 
[pr-rule]   - Have a near term view of implementation and a long term vision. In other words, start small and but keep in mind on where the app is heading down the road. All of the app's code goes in a root folder named `app`. All content is 1 feature per file. Each controller, service, module, view is in its own file. All 3rd party vendor scripts are stored in another root folder and not in the `app` folder. I didn't write them and I don't want them cluttering my app (`bower_components`, `scripts`, `lib`).
[pr-text]     Note: Find more details and reasoning behind the structure at [this original post on application structure](http://www.johnpapa.net/angular-app-structuring-guidelines/).

[h3] ### Layout
[h5] Y151
[pr-text] 
[pr-rule]   - Place components that define the overall layout of the application in a folder named `layout`. These may include a shell view and controller may act as the container for the app, navigation, menus, content areas, and other regions. 
[pr-why]     *Why?*: Organizes all layout in a single place re-used throughout the application.
[pr-text]

[pr-why]     *Why?*: The LIFT guidelines are all covered.
[pr-text] 
[pr-why]     *Why?*: Helps reduce the app from becoming cluttered through organizing the content and keeping them aligned with the LIFT guidelines.

[pr-why]     *Why?*: When there are a lot of files (10+) locating them is easier with a consistent folder structures and more difficult in flat structures.

[h2] ## Modularity
[h3] ### Many Small, Self Contained Modules
[h5] Y160
[pr-text] 
[pr-rule]   - Create small modules that encapsulate one responsibility.
[pr-why]     *Why?*: Modular applications make it easy to plug and go as they allow the development teams to build vertical slices of the applications and roll out incrementally. This means we can plug in new features as we develop them.
[pr-text] 
[h3] ### Create an App Module
[h5] Y161
[pr-text] 
[pr-rule]   - Create an application root module whose role is pull together all of the modules and features of your application. Name this for your application.
[pr-why]     *Why?*: AngularJS encourages modularity and separation patterns. Creating an application root module whose role is to tie your other modules together provides a very straightforward way to add or remove modules from your application.
[pr-text] 
[h3] ### Keep the App Module Thin
[h5] Y162
[pr-text] 
[pr-rule]   - Only put logic for pulling together the app in the application module. Leave features in their own modules.
[pr-why]     *Why?*: Adding additional roles to the application root to get remote data, display views, or other logic not related to pulling the app together muddies the app module and make both sets of features harder to reuse or turn off.
[pr-text] 
[pr-why]     *Why?*: The app module becomes a manifest that describes which modules help define the application. 
[pr-text] 
[h3] ### Feature Areas are Modules
[h5] Y163
[pr-text] 
[pr-rule]   - Create modules that represent feature areas, such as layout, reusable and shared services, dashboards, and app specific features (e.g. customers, admin, sales).
[pr-why]     *Why?*: Self contained modules can be added to the application with little or no friction.
[pr-text] 
[pr-why]     *Why?*: Sprints or iterations can focus on feature areas and turn them on at the end of the sprint or iteration.
[pr-text] 
[pr-why]     *Why?*: Separating feature areas into modules makes it easier to test the modules in isolation and reuse code. 
[pr-text] 
[h3] ### Reusable Blocks are Modules
[h5] Y164
[pr-text] 
[pr-rule]   - Create modules that represent reusable application blocks for common services such as exception handling, logging, diagnostics, security, and local data stashing.
[pr-why]     *Why?*: These types of features are needed in many applications, so by keeping them separated in their own modules they can be application generic and be reused across applications.
[pr-text] 
[h3] ### Module Dependencies
[h5] Y165
[pr-text] 
[pr-rule]   - The application root module depends on the app specific feature modules and any shared or reusable modules.
[pr-text]     ![Modularity and Dependencies](https://raw.githubusercontent.com/johnpapa/angularjs-styleguide/master/assets/modularity-1.png)
[pr-text] 
[pr-why]     *Why?*: The main app module contains a quickly identifiable manifest of the application's features. 
[pr-text] 
[pr-why]     *Why?*: Each feature area contains a manifest of what it depends on, so it can be pulled in as a dependency in other applications and still work. 
[pr-text] 
[pr-why]     *Why?*: Intra-App features such as shared data services become easy to locate and share from within `app.core` (choose your favorite name for this module).
[pr-text] 
[pr-text]     Note: This is a strategy for consistency. There are many good options here. Choose one that is consistent, follows AngularJS's dependency rules, and is easy to maintain and scale.
[pr-text] 
[pr-text]     > My structures vary slightly between projects but they all follow these guidelines for structure and modularity. The implementation may vary depending on the features and the team. In other words, don't get hung up on an exact like-for-like structure but do justify your structure using consistency, maintainability, and efficiency in mind. 
[pr-text] 
[pr-text]     > In a small app, you can also consider putting all the shared dependencies in the app module where the feature modules have no direct dependencies. This makes it easier to maintain the smaller application, but makes it harder to reuse modules outside of this application.
[pr-text] 
[pr-text] 
[h2] ## Startup Logic
[h3] ### Configuration
[h5] Y170
[pr-text] 
[pr-rule]   - Inject code into [module configuration](https://docs.angularjs.org/guide/module#module-loading-dependencies) that must be configured before running the angular app. Ideal candidates include providers and constants.
[pr-why]     *Why?*: This makes it easier to have a less places for configuration.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   angular
[pr-text]       .module('app')

[pr-text]   configure.$inject =

[pr-text]       exceptionHandlerProvider.configure(config.appErrorPrefix);
[pr-text]       configureStateHelper();
[pr-text] 
[pr-text]       toastr.options.timeOut = 4000;
[pr-text]       toastr.options.positionClass = 'toast-bottom-right';
[pr-text] 
[pr-text]       ////////////////
[pr-text] 
[pr-text]       function configureStateHelper() {
[pr-text]           routerHelperProvider.configure({
[pr-text]               docTitle: 'NG-Modular: '
[pr-text]           });
[pr-text]       }
[pr-text]   }
[pr-text]   ```

[h3] ### Run Blocks
[h5] Y171
[pr-text] 
[pr-rule]   - Any code that needs to run when an application starts should be declared in a factory, exposed via a function, and injected into the [run block](https://docs.angularjs.org/guide/module#module-loading-dependencies).
[pr-why]     *Why?*: Code directly in a run block can be difficult to test. Placing in a factory makes it easier to abstract and mock.
[pr-text] 
[pr-text]   ```javascript
[pr-text]   angular
[pr-text]       .module('app')
[pr-text]       .run(runBlock);
[pr-text] 
[pr-text]     runBlock.$inject = ['authenticator', 'translator'];
[pr-text] 
[pr-text]     function runBlock(authenticator, translator) {
[pr-text]         authenticator.initialize();
[pr-text]         translator.initialize();
[pr-text]     }
[pr-text]   ```

[h2] ## Angular $ Wrapper Services
[h3] ### $document and $window
[h5] Y180
[pr-text] 
[pr-rule]   - Use [`$document`](https://docs.angularjs.org/api/ng/service/$document) and [`$window`](https://docs.angularjs.org/api/ng/service/$window) instead of `document` and `window`.
[pr-why]     *Why?*: These services are wrapped by Angular and more easily testable than using document and window in tests. This helps you avoid having to mock document and window yourself.
[pr-text] 
[h3] ### $timeout and $interval
[h5] Y181
[pr-text] 
[pr-rule]   - Use [`$timeout`](https://docs.angularjs.org/api/ng/service/$timeout) and [`$interval`](https://docs.angularjs.org/api/ng/service/$interval) instead of `setTimeout` and `setInterval` .
[pr-why]     *Why?*: These services are wrapped by Angular and more easily testable and handle AngularJS's digest cycle thus keeping data binding in sync.

[h3] ### Organizing Tests
[h5] Y197
[pr-text] 
[pr-rule]   - Place unit test files (specs) side-by-side with your client code. Place specs that cover server integration or test multiple components in a separate `tests` folder.
[pr-why]     *Why?*: Unit tests have a direct correlation to a specific component and file in source code. 
[pr-text] 
[pr-why]     *Why?*: It is easier to keep them up to date since they are always in sight. When coding whether you do TDD or test during development or test after development, the specs are side-by-side and never out of sight nor mind, and thus more likely to be maintained which also helps maintain code coverage.
[pr-text] 
[pr-why]     *Why?*: When you update source code it is easier to go update the tests at the same time.
[pr-text] 
[pr-why]     *Why?*: Placing them side-by-side makes it easy to find them and easy to move them with the source code if you move the source.
[pr-text] 
[pr-why]     *Why?*: Having the spec nearby makes it easier for the source code reader to learn how the component is supposed to be used and to discover its known limitations.
[pr-text] 
[pr-why]     *Why?*: Separating specs so they are not in a distributed build is easy with grunt or gulp.
[pr-text] 
[pr-text]     ```
[pr-text]     /src/client/app/customers/customer-detail.controller.js
[pr-text]                              /customer-detail.controller.spec.js
[pr-text]                              /customers.controller.spec.js
[pr-text]                              /customers.controller-detail.spec.js
[pr-text]                              /customers.module.js
[pr-text]                              /customers.route.js
[pr-text]                              /customers.route.spec.js
[pr-text]     ```

[h2] ## Comments
[h3] ### jsDoc
[h5] Y220
[pr-text] 
[pr-rule]   - If planning to produce documentation, use [`jsDoc`](http://usejsdoc.org/) syntax to document function names, description, params and returns. Use `@namespace` and `@memberOf` to match your app structure.
[pr-why]     *Why?*: You can generate (and regenerate) documentation from your code, instead of writing it from scratch.
[pr-text] 
[pr-why]     *Why?*: Provides consistency using a common industry tool.
[pr-text] 
[pr-text]     ```javascript
[pr-text]     /**
[pr-text]      * Logger Factory
[pr-text]      * @namespace Factories
[pr-text]      */
[pr-text]     (function() {
[pr-text]       angular
[pr-text]           .module('app')
[pr-text]           .factory('logger', logger);
[pr-text] 
[pr-text]       /**
[pr-text]        * @namespace Logger
[pr-text]        * @desc Application wide logger
[pr-text]        * @memberOf Factories
[pr-text]        */
[pr-text]       function logger($log) {
[pr-text]           var service = {
[pr-text]              logError: logError
[pr-text]           };
[pr-text]           return service;
[pr-text] 
[pr-text]           ////////////
[pr-text] 
[pr-text]           /**
[pr-text]            * @name logError
[pr-text]            * @desc Logs errors
[pr-text]            * @param {String} msg Message to log 
[pr-text]            * @returns {String}
[pr-text]            * @memberOf Factories.Logger
[pr-text]            */
[pr-text]           function logError(msg) {
[pr-text]               var loggedMsg = 'Error: ' + msg;
[pr-text]               $log.error(loggedMsg);
[pr-text]               return loggedMsg;
[pr-text]           };
[pr-text]       }
[pr-text]     })();
[pr-text]     ```

[h2] ## Constants
[h3] ### Vendor Globals
[h5] Y240
[pr-text] 
[pr-rule]   - Create an AngularJS Constant for vendor libraries' global variables.
[pr-why]     *Why?*: Provides a way to inject vendor libraries that otherwise are globals. This improves code testability by allowing you to more easily know what the dependencies of your components are (avoids leaky abstractions). It also allows you to mock these dependencies, where it makes sense.
[pr-text] 
[pr-text]     ```javascript
[pr-text]     // constants.js
[pr-text] 
[pr-text]     /* global toastr:false, moment:false */
[pr-text]     (function() {
[pr-text]         'use strict';
[pr-text] 
[pr-text]         angular
[pr-text]             .module('app.core')
[pr-text]             .constant('toastr', toastr)
[pr-text]             .constant('moment', moment);
[pr-text]     })();
[pr-text]     ```
[pr-text]     
[h5] Y241
[pr-text] 
[pr-rule]   - Use constants for values that do not change and do not come from another service. When constants are used only for a module that may be reused in multiple applications, place constants in a file per module named after the module. Until this is required, keep constants in the main module in a `constants.js` file.
[pr-why]     *Why?*: A value that may change, even infrequently, should be retrieved from a service so you do not have to change the source code. For example, a url for a data service could be placed in a constants but a better place would be to load it from a web service.
[pr-text] 
[pr-why]     *Why?*: Constants can be injected into any angular component, including providers.
[pr-text] 
[pr-why]     *Why?*: When an application is separated into modules that may be reused in other applications, each stand-alone module should be able to operate on its own including any dependent constants. 
[pr-text] 
[pr-text]     ```javascript
[pr-text]     // Constants used by the entire app
[pr-text]     angular
[pr-text]         .module('app.core')
[pr-text]         .constant('moment', moment);
[pr-text] 
[pr-text]     // Constants used only by the sales module
[pr-text]     angular
[pr-text]         .module('app.sales')
[pr-text]         .constant('events', {
[pr-text]             ORDER_CREATED: 'event_order_created',
[pr-text]             INVENTORY_DEPLETED: 'event_inventory_depleted'
[pr-text]         });
[pr-text]     ```

[h2] ## Routing
[h2-note] Client-side routing is important for creating a navigation flow between views and composing views that are made of many smaller templates and directives.
[h5] Y270
[h3] 
[pr-text] 
[pr-rule]   - Use the [AngularUI Router](http://angular-ui.github.io/ui-router/) for client-side routing.
[pr-why]     *Why?*: UI Router offers all the features of the Angular router plus a few additional ones including nested routes and states.
[pr-text] 
[pr-why]     *Why?*: The syntax is quite similar to the Angular router and is easy to migrate to UI Router.
[pr-text] 
[h5] Y271
[pr-text] 
[pr-rule]   - Define routes for views in the module where they exist. Each module should contain the routes for the views in the module.
[pr-why]     *Why?*: Each module should be able to stand on its own.
[pr-text] 
[pr-why]     *Why?*: When removing a module or adding a module, the app will only contain routes that point to existing views.
[pr-text] 
[pr-why]     *Why?*: This makes it easy to enable or disable portions of an application without concern over orphaned routes.
[pr-text]

[h2] ## Task Automation
[h2-note] Use [Gulp](http://gulpjs.com) or [Grunt](http://gruntjs.com) for creating automated tasks. Gulp leans to code over configuration while Grunt leans to configuration over code. I personally prefer Gulp as I feel it is easier to read and write, but both are excellent.
[h5] Y400
[h3] 
[pr-text] 
[pr-rule]   - Use task automation to list module definition files `*.module.js` before all other application JavaScript files.
[pr-why]     *Why?*: Angular needs the module definitions to be registered before they are used.
[pr-text] 
[pr-why]     *Why?*: Naming modules with a specific pattern such as `*.module.js` makes it easy to grab them with a glob and list them first.
[pr-text] 
[pr-text]     ```javascript
[pr-text]     var clientApp = './src/client/app/';
[pr-text] 
[pr-text]     // Always grab module files first
[pr-text]     var files = [
[pr-text]       clientApp + '**/*.module.js',
[pr-text]       clientApp + '**/*.js'
[pr-text]     ];
[pr-text]     ```
